{# ─── QUICK START · CLI ─── #}
Quick start
Using the command-line CLI is the recommended and easiest path to contribute your test runs or your own tests. Follow this steps:
  1. 01
    Fork the repo on GitHub
  2. 02
    Clone it
    git clone https://github.com/your_github_username/agent-arena.git
cd agent-arena
  3. 03
    Run the cli
    ./agent-arenacli.py
    You'll be presented with the available options:
    run add ./agent-arena-cli.py run add A wizard to contribute a run of one of the available tests
    test add ./agent-arena-cli.py test add A wizard to contribute an entirely new test of your own
    validate ./agent-arena-cli.py validate Checks for errors in yaml files after manual edits
    browse ./agent-arena-cli.py browse Text interface for browsing tests and runs
  4. 04
    After adding your contribution, commit it to your forked repo
    # Commit your changes locally
git add .
git commit -m "Add my test contribution"

# Push the changes to your forked repository
git push origin main
  5. 05
    Create a push request in your forked repository's GitHub page to send it for approval.
{# ─── HOW TO RUN A TEST ─── #}
How to run a test
Find the test you want to run in the tests section, you'll find the prompts you need to feed to your agent there.
  • Run each stage in order
  • If you're using additional MCP servers, skills, custom subagents, hooks, or similar beyond the agent's defaults, document it by adding them to settings (e.g. mcps=linear,sentry / skills=simplify,review).
  • Copy and paste each stage prompt exactly as written.
  • Once a stage finishes, take note of the usage and close your agent.
  • Each stage should be run after a fresh start of your agent, on the code left by the previous stage.
{# ─── HOW TO RECORD TIME ─── #}
How to record time
Record API time: The time the agent actually spent calling the model. Do not include time you spent reading responses, approving tool confirmations, sipping coffee, etc. For example, in Claude Code, /usage shows exactly this. When not possible, please estimate as best as possible and mention it in the notes.
{# ─── PERSONAL URL ─── #}
Your personal URL
The URL you specify as contributor_url (Personal URL) is your unique ID. Always use the exact same URL across every contribution so the leaderboard can group your runs and accumulate your ranking. Also, if you use your GitHub profile URL, your avatar shows automatically.
{# ─── OPTIONALLY ADD CODE RESULTS ─── #}
Optionally add code results
If you feel like it, you can add the code results as your agent generated them to your contribution.
Add the resulting code of each stage to /tests/<test name>/results/<run-id>/. One subdirectory per stage you ran, named exactly like the stage id (e.g. stage-1-first-run). Inside each, put the complete source code the LLM produced, even if most of it is duplicated from earlier stages.
{# ─── MODEL IDENTIFIER ─── #}
the model identifier
Use the official identifier so runs of the same model can be grouped on the leaderboard.
closed / hosted Use the canonical name from the provider's docs, e.g. sonnet-4.6, gpt-5-mini, gemini-2.5-pro.

unsure? For a cross-provider catalog, openrouter.ai/models is a convenient reference.

Without the CLI · contribute a run manually
You can always add your runs and tests by manually adding YAML files instead of letting the CLI create them for you. Run ./agent-arena-cli.py validate after any manual edit to check for errors.
  1. 01
    pick a test
    Use the tests section or browse /tests to find the test you want to run.
  2. 02
    create a run directory
    Place it at /tests/<test name>/results/<run-id>/. The run id is a short, unique slug — typically <your-github-username>-<agent>-<model>-<settings>, e.g. tin-cat-claude-code-sonnet-4.6-high-effort. For multiple runs of the same config, append -2 or a date.
  3. 03
    optionally drop in each stage's resulting code
    One subdirectory per stage you ran, named exactly like the stage id in test.yaml (e.g. stage-1-first-run). Inside each, put the complete source code the LLM produced — even if most of it is duplicated from earlier stages.
  4. 04
    add a run.yaml file
    A single manifest at the root of your run directory. Only include stages you actually ran, no empty placeholders. Here's the schema:
    {{ run_yaml_example }}
  5. 06
    open a PR
    Fork, branch, push, open the PR against main. Get into the leaderboard as soon as it lands.
